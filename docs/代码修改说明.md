## 详细修改说明

### 1. server.h — 新增 `rename_path` 声明

**位置**：`server/src/server.h`

**原代码**：全局变量区无 `rename_path`

**修改后**：
```c
char rename_path[1000];   /* 新增：RNFR/RNTO 用于保存重命名源路径 */
```

**原因**：`server.c` 的 `ftp_server_response()` 函数在处理 `RNFR` 和 `RNTO` 时直接使用了 `rename_path` 变量，但该变量在 `server.h` 和 `server.c` 中均未声明。原代码中有一行注释掉的声明 `//char rename_path[100];`，说明作者曾计划将其作为局部变量，但后来改为全局使用却忘记补声明。这导致编译时产生 `undeclared identifier` 错误，程序无法编译。

---

### 2. server.c — 新增 `get_pwd()` 实现

**位置**：`server/src/server.c` 文件顶部

**原代码**：`server.h` 中有声明 `void get_pwd(int socket, char *info);`，但 `server.c` 中无任何实现。`ftp_server_response()` 的 `PWD` 分支调用了 `get_pwd(client_sock, info)`。

**新增代码**：
```c
void get_pwd(int socket, char *info) {
    char path[1000];
    getcwd(path, sizeof(path));
    sprintf(info, "257 \"%s\" is current directory.", path);
}
```

**原因**：缺少实现会导致链接阶段报 `undefined reference to 'get_pwd'` 错误，程序无法链接。

---

### 3. server.c — 新增 `get_list_server()` 实现

**位置**：`server/src/server.c` 文件顶部

**原代码**：`server.h` 中有声明，`LIST` 命令处理调用了 `get_list_server(file_connect_sock)`，但无实现。

**新增代码**：
```c
void get_list_server(const int sock) {
    char buffer[MAX_SIZE];
    FILE *fp = popen("ls -l", "r");
    if (fp == NULL) {
        const char *err = "ls failed\r\n";
        send(sock, err, strlen(err), 0);
        return;
    }
    int n;
    while ((n = fread(buffer, 1, MAX_SIZE, fp)) > 0) {
        send(sock, buffer, n, 0);
    }
    pclose(fp);
}
```

**原因**：同上，链接错误。使用 `popen("ls -l", "r")` 执行系统命令并将结果通过数据连接发送给客户端。

---

### 4. server.c — 修复 `RNFR` 命令逻辑

**位置**：`server/src/server.c`，`ftp_server_response()` 中 RNFR 分支

**原代码**：
```c
else if(strcmp(client_cmd, "RNFR") == 0){
    char info[100];
    char control[200];
    strcpy(rename_path, cur_path);
    strcat(rename_path, client_arg);
    sprintf(control, "mv %s %s", rename_path, rename_path);  // 源和目标相同！
    int r = system(control);
    if(r == 0){
        sprintf(info, "350 ...");
        send_infomation(client_sock, info);
    }
    ...
}
```

**修改后**：
```c
else if(strcmp(client_cmd, "RNFR") == 0){
    char info[200];
    strcpy(rename_path, cur_path);
    strcat(rename_path, client_arg);
    if(access(rename_path, F_OK) == 0){   // 用 access() 检查文件是否存在
        sprintf(info, "350 \"%s\": File exists, ready for destination name.", client_arg);
        send_infomation(client_sock, info);
    }
    else{
        sprintf(info, "550 \"%s\": no such file or directory.", client_arg);
        send_infomation(client_sock, info);
        memset(rename_path, 0, sizeof(rename_path));  // 清空无效路径
    }
}
```

**原因**：原代码执行 `mv X X`（源路径和目标路径相同），在大多数 Linux 系统上会报错（`mv: 'X' and 'X' are the same file`），导致 `system()` 返回非零值，RNFR 总是返回 550 失败。RFC 959 规定 RNFR 的语义是检查源文件是否存在并锁定，应使用 `access(path, F_OK)` 检查文件存在性。

### 5. server.c — 新增 `DELE` 命令处理

**位置**：`server/src/server.c`，`ftp_server_response()` RMD 分支之前

**原代码**：无 DELE 分支，收到 DELE 命令会走到 `else` 分支返回 `502 Invalid command`。

**新增代码**：
```c
else if(strcmp(client_cmd, "DELE") == 0){
    char info[200];
    char path[200];
    strcpy(path, cur_path);
    strcat(path, client_arg);
    if(remove(path) == 0){
        sprintf(info, "250 \"%s\": file deleted.", path);
        send_infomation(client_sock, info);
    }
    else{
        sprintf(info, "550 \"%s\": no such file or permission denied.", path);
        send_infomation(client_sock, info);
    }
}
```

**原因**：简历中提到支持 `DELE` 命令，`server.h` 中也定义了 `#define DELE 15`，客户端也实现了 DELE 的发送逻辑，但服务端缺少对应处理。使用标准 C 库函数 `remove()` 删除文件。

### 6. server.c — 新增 `CDUP` 命令处理

**位置**：`server/src/server.c`，CWD 分支之后

**原代码**：无 CDUP 分支。

**新增代码**：
```c
else if(strcmp(client_cmd, "CDUP") == 0){
    char info[200];
    memset(info, 0, 200);
    if(chdir("..") >= 0){
        getcwd(cur_path, 1000);
        sprintf(info, "257 CDUP command successful \"%s\" is current directory.", cur_path);
        strcat(cur_path, "/");
        send_infomation(client_sock, info);
    }
    else{
        send_infomation(client_sock, "550 CDUP failed.");
    }
}
```

**原因**：客户端实现了 CDUP 命令发送，`server.h` 中定义了 `#define CDUP 13`，服务端却未处理。CDUP 等价于 `CWD ..`，调用 `chdir("..")` 实现。

### 7. server.c — `get_file_size` 添加 `fclose`

**位置**：`server/src/server.c`，`get_file_size()` 函数

**原代码**：
```c
int get_file_size(char *filename){
    FILE *f = fopen(filename, "r");
    if(!f) return -1;
    fseek(f, 0L, SEEK_END);
    int file_size = ftell(f);
    return file_size;  // 未调用 fclose，文件描述符泄漏
}
```

**修改后**：
```c
int get_file_size(char *filename){
    FILE *f = fopen(filename, "r");
    if(!f) return -1;
    fseek(f, 0L, SEEK_END);
    int file_size = ftell(f);
    fclose(f);   // 新增
    return file_size;
}
```

**原因**：每次 RETR/STOR 都调用此函数，不 `fclose` 会导致文件描述符逐渐耗尽（每个进程有文件描述符上限）。

### 9. client.c — 修复 `RNTO` 命令字符串错误

**位置**：`client/src/client.c`，`client_loop()` 函数

**原代码**：
```c
else if(strcmp(client_cmd, "Rq") == 0){    // 拼写错误！
    client_status = RNTO;
    send_infomation(client_sock, sentence);
    recv_response(reply);
}
```

**修改后**：
```c
else if(strcmp(client_cmd, "RNTO") == 0){  // 正确的 FTP 命令名
    client_status = RNTO;
    send_infomation(client_sock, sentence);
    recv_response(reply);
}
```

**原因**：`"Rq"` 明显是笔误（可能是调试时的临时字符串），导致客户端无法发送 RNTO 命令，重命名功能完全失效。

---

### 10. client.c — 删除忙等待循环

**位置**：`client/src/client.c`，`client_loop()` 函数开头

**原代码**：
```c
int client_loop() {
    int len;
    char sentence[MAX_SIZE];
    char reply[MAX_SIZE];
    for(int p = 0; p < 100000000; p++);  // 忙等待！
    printf("ftp client> ");
    ...
}
```

**修改后**：直接删除该行。

**原因**：循环 1 亿次的忙等待（busy-wait）会在每次等待用户输入前浪费大量 CPU 时间，实际运行会感觉明显卡顿（约几秒钟）。这是开发时的调试残留代码，`fgets()` 本身已经是阻塞调用，完全不需要额外延迟。

## 简历描述与代码的对应关系

简历中写道：

> 基于 Linux socket 接口和 C 语言实现 FTP 客户端与服务端，支持 USER、MODE、CWD、LIST、RNFR 等标准指令

**说明**：
- `MODE` 命令在代码中实际为 `TYPE`（设置传输类型），FTP 协议中 `MODE` 和 `TYPE` 是不同命令，代码实现的是 `TYPE`。简历中写 `MODE` 可改为 `TYPE` 或 `PASV` 更准确。
- `RNFR` 需配合 `RNTO` 使用（rename from / rename to），可以写成 `RNFR/RNTO`。
- 实际支持的完整命令集远超简历列举，包括：`USER`、`PASS`、`SYST`、`TYPE`、`PORT`、`PASV`、`RETR`、`STOR`、`LIST`、`PWD`、`CWD`、`CDUP`、`MKD`、`RMD`、`DELE`、`RNFR`、`RNTO`、`QUIT`、`ABOR`。

> 采用多进程模型处理并发连接，实现控制连接与数据连接的分离，保证文件传输的可靠性与会话隔离

这一描述与代码完全吻合：
- `fork()` 多进程 ✓
- 控制/数据连接分离（两套 socket）✓
- TCP 保证可靠性 ✓
- 进程地址空间隔离保证会话独立 ✓
